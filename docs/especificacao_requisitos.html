1. Visão Geral do Sistema

O Sistema Segue-me é uma aplicação web desenvolvida para gerenciar de forma completa e integrada os Encontros de Jovens, movimento de evangelização católica que realiza retiros espirituais para jovens.

O sistema oferece funcionalidades abrangentes para registro de participantes (encontristas), organização de equipes, gestão de trabalhadores voluntários, controle financeiro de contribuições e despesas, programação de palestras, geração de relatórios gerenciais e emissão de fichas digitais para acompanhamento dos encontristas durante o evento.

Nota: O termo "Segue-me" faz referência ao chamado de Jesus Cristo aos discípulos ("Segue-me e eu farei de vós pescadores de homens"), representando o convite ao jovem para seguir o caminho de Cristo através da participação no encontro de evangelização.

2. Objetivos do Sistema

2.1. Objetivo Geral

Fornecer uma plataforma integrada e centralizada para gerenciar todos os aspectos relacionados à organização, execução e acompanhamento de Encontros de Jovens, desde o cadastro de participantes até a gestão financeira e geração de relatórios pós-evento.

2.2. Objetivos Específicos

  • Centralizar o cadastro de pessoas (encontristas, trabalhadores, palestrantes, dirigentes) em uma única base de dados;
  • Facilitar o processo de inscrição e acompanhamento de encontristas;
  • Organizar equipes de trabalho e distribuir responsabilidades entre trabalhadores;
  • Gerenciar a estrutura hierárquica de dirigentes através de pastas organizacionais;
  • Controlar contribuições financeiras dos trabalhadores e despesas do encontro;
  • Programar e acompanhar a execução de palestras durante o encontro;
  • Gerar relatórios gerenciais para tomada de decisão;
  • Emitir fichas digitais individualizadas para acompanhamento pastoral dos encontristas;
  • Permitir vendas de artigos religiosos durante o evento;
  • Manter histórico completo de todos os encontros realizados;
  • Garantir a segurança e integridade dos dados pessoais dos participantes.

3. Descrição do Domínio de Negócio

3.1. Contexto do Encontro de Jovens

Os Encontros de Jovens são retiros espirituais católicos realizados periodicamente (geralmente anualmente) com duração de um final de semana (3 dias). Possuem estrutura organizada com:

  • Encontristas (Seguimistas): Jovens que participam do encontro pela primeira vez ou como retornantes;
  • Equipe de Trabalho: Voluntários que organizam e conduzem as atividades do encontro;
  • Dirigentes: Membros responsáveis pela coordenação geral e tomada de decisões estratégicas;
  • Palestrantes: Pessoas convidadas para ministrar palestras sobre temas de fé;
  • Círculos: Pequenos grupos de partilha formados durante o encontro para discussão das palestras;
  • Pastas: Divisões organizacionais que agrupam equipes com objetivos similares.

3.2. Ciclo de Vida do Encontro

  1. Planejamento: Definição de datas, local, tema, capacidade e valores;
  2. Formação de Equipe: Convite e organização de trabalhadores em equipes especializadas;
  3. Inscrições: Recebimento de inscrições de encontristas com pagamento;
  4. Preparação: Realização de reuniões preparatórias, ensaios e organização logística;
  5. Execução: Realização do encontro com palestras, dinâmicas, celebrações e partilhas;
  6. Encerramento: Celebração final, avaliações e fechamento financeiro;
  7. Pós-Encontro: Acompanhamento espiritual dos encontristas através de grupos de perseverança.

3.3. Estrutura Organizacional

A organização típica de um Encontro de Jovens segue a seguinte hierarquia:

  • Coordenação Geral: Dirigentes responsáveis pela direção do movimento;
  • Pastas: Divisões organizacionais (ex: Pasta de Formação, Pastoral, Logística, Financeira);
  • Equipes: Grupos de trabalho específicos (ex: Equipe de Cozinha, Cânticos, Decoração, Secretaria);
  • Trabalhadores: Voluntários que executam as atividades de cada equipe.

5. Requisitos Funcionais

5.1. Gestão de Pessoas e Casais

RF001 - Cadastrar Pessoa

Descrição: O sistema deve permitir o cadastro completo de pessoas físicas.

Dados:

  • Nome completo (obrigatório, 3-100 caracteres)
  • CPF (opcional, único, formato válido)
  • Data de nascimento (obrigatório, data passada)
  • Sexo (M/F)
  • Endereço (opcional, até 200 caracteres)
  • Telefone (único, formato: (00) 00000-0000)
  • E-mail (opcional, formato válido)
  • Escolaridade (enum: FUNDAMENTAL, MEDIO, SUPERIOR, POS_GRADUACAO)
  • Sacramento (enum: BATISMO, CRISMA, EUCARISTIA)
  • Observações (texto livre)
  • Foto (upload de imagem)
  • Status ativo/inativo

Prioridade: ALTA

RF002 - Consultar Pessoa

Descrição: O sistema deve permitir consultar pessoas por diversos critérios.

Filtros: Nome, CPF, telefone, e-mail, status

Prioridade: ALTA

RF003 - Editar Pessoa

Descrição: O sistema deve permitir editar dados de pessoas cadastradas.

Prioridade: ALTA

RF004 - Inativar Pessoa

Descrição: O sistema deve permitir marcar pessoas como inativas (exclusão lógica).

Prioridade: MÉDIA

RF005 - Cadastrar Casal

Descrição: O sistema deve permitir cadastrar casais relacionando duas pessoas.

Dados: Pessoa 1, Pessoa 2, Data de casamento, Observações, Foto do casal

Regra: A combinação (Pessoa1, Pessoa2) deve ser única

Prioridade: MÉDIA

RF006 - Gestão de Fotos

Descrição: O sistema deve permitir upload, visualização e exclusão de fotos de pessoas.

Formatos: JPG, PNG, GIF (máx. 5MB)

Armazenamento: Diretório configurável via propriedade caminho_fotos

Prioridade: MÉDIA

5.2. Transferências entre Paróquias

RF007 - Registrar Transferência de Saída

Descrição: O sistema deve permitir registrar a transferência de um seguidor desta paróquia para outra.

Dados:

  • Pessoa (obrigatório, seleção de seguidor ativo)
  • Paróquia de destino (obrigatório, até 150 caracteres)
  • Observações (opcional, até 500 caracteres)
  • Data de solicitação (auto-preenchida)

Efeitos:

  • Pessoa passa para status TRANSFERIDO_SAIDA
  • Pessoa é marcada como inativa automaticamente
  • Registro histórico é criado na tabela transferencia
  • Seguidor transferido não pode trabalhar em encontros desta paróquia

Regra de Negócio: Apenas seguidores com status ATIVO_ORIGEM ou RETORNADO podem ser transferidos

Prioridade: ALTA

RF008 - Registrar Aceitação de Entrada

Descrição: O sistema deve permitir aceitar um seguidor vindo de outra paróquia.

Dados:

  • Pessoa (nova ou existente)
  • Paróquia de origem (obrigatório, até 150 caracteres)
  • Observações (opcional, até 500 caracteres)
  • Data de solicitação (auto-preenchida)

Efeitos:

  • Pessoa passa para status TRANSFERIDO_ENTRADA
  • Pessoa é marcada como ativa automaticamente
  • Campo paroquiaOrigem é preenchido com a paróquia de origem
  • Registro histórico é criado na tabela transferencia
  • Seguidor aceito pode trabalhar em encontros desta paróquia

Prioridade: ALTA

RF009 - Registrar Retorno à Origem

Descrição: O sistema deve permitir registrar o retorno de um seguidor que havia saído para outra paróquia.

Dados:

  • Pessoa (obrigatório, apenas com status TRANSFERIDO_SAIDA)
  • Observações (opcional, até 500 caracteres)
  • Data de solicitação (auto-preenchida)

Efeitos:

  • Pessoa passa para status RETORNADO
  • Pessoa é marcada como ativa automaticamente
  • Sistema identifica automaticamente a paróquia de onde está retornando (última transferência de saída)
  • Registro histórico é criado na tabela transferencia
  • Seguidor que retornou pode trabalhar em encontros novamente

Regra de Negócio: Apenas seguidores com status TRANSFERIDO_SAIDA podem retornar

Algoritmo de Detecção: O sistema busca o último registro de TRANSFERIDO_SAIDA ordenado por data e ID, e usa o campo paroquiaDestino como origem do retorno

Prioridade: ALTA

5.3.1. Cronograma de Atividades

RF014 - Cadastrar Atividade no Cronograma

Descrição: O sistema deve permitir cadastrar atividades no cronograma de um encontro.

Dados:

  • Encontro (obrigatório, somente encontros ativos)
  • Título (obrigatório, até 200 caracteres)
  • Tipo de Atividade (obrigatório, enum: PALESTRA, DINAMICA, REFEICAO, ORACAO, SILENCIO, INTERVALO, LOUVOR, TESTEMUNHO, ENCERRAMENTO, OUTRO)
  • Data/Hora de Início (obrigatório)
  • Data/Hora de Fim (opcional)
  • Responsável (opcional, até 100 caracteres)
  • Local da Atividade (opcional, até 200 caracteres)
  • Descrição (opcional, até 500 caracteres)
  • Ordem (opcional, inteiro para ordenação manual)
  • Palestra Vinculada (opcional, somente quando tipo = PALESTRA)

Prioridade: ALTA

RF015 - Visualizar Cronograma por Encontro

Descrição: O sistema deve exibir o cronograma de um encontro em formato de linha do tempo visual.

Funcionalidades:

  • Agrupamento por dia (separação automática por data)
  • Ordenação cronológica por data/hora de início
  • Exibição colorida conforme tipo de atividade (código de cores definido no enum)
  • Exibição de ícone, duração calculada e palestrantes vinculados (quando tipo PALESTRA)
  • Filtro por encontro (ativos e finalizados)
  • Indicação "(Realizado)" para encontros já finalizados
  • Ações de Nova, Editar e Excluir atividade disponíveis apenas para encontros ativos

Prioridade: ALTA

RF016 - Vincular Palestra ao Cronograma

Descrição: Quando o tipo de atividade for PALESTRA, o sistema deve permitir vincular uma palestra cadastrada do mesmo encontro à atividade do cronograma.

Comportamento:

  • Campo de seleção de palestra aparece dinamicamente ao escolher tipo PALESTRA
  • Lista de palestras é filtrada pelo encontro selecionado
  • A vinculação é opcional (palestra pode ser deixada em branco)
  • Quando vinculada, o cronograma exibe o nome do tema, título da palestra e nome dos palestrantes

Prioridade: MÉDIA

5.10.1. Análise Avançada

RF085 - Análise Avançada de Encontristas

Descrição: O sistema deve fornecer uma aba de "Análise Avançada" no painel de relatórios com indicadores gerenciais estratégicos.

Indicadores:

  • Distribuição Etária: Agrupamento por faixas (<16, 16-17, 18-20, 21-23, >23, sem idade), com destaque visual (⚠) para encontristas fora da faixa permitida de 16-23 anos
  • Conformidade de Faixa Etária: Percentual dentro da faixa 16-23 anos e contagem de fora da faixa
  • Conversão Encontrista→Trabalhador por Encontro: Percentual de encontristas que se tornaram trabalhadores em encontros subsequentes
  • Crescimento de Inscritos: Comparação entre o último e o penúltimo encontro, com indicação positiva/negativa
  • Dados de Saúde: Total de encontristas com algum dado de saúde preenchido
  • Menores e Responsável: Total de menores de 18 anos e quantos possuem dados de responsável preenchidos

Regra de Faixa Etária: A faixa etária permitida para encontristas é 16 a 23 anos. Encontristas fora desta faixa são permitidos tecnicamente, mas sinalizados como exceção nos relatórios.

Prioridade: MÉDIA

7.2. Relacionamentos Principais

  • Pessoa → Encontrista: Uma pessoa pode participar como encontrista em vários encontros (1:N)
  • Encontro → Encontrista: Um encontro possui vários encontristas (1:N)
  • Pessoa → Trabalhador: Uma pessoa pode ser trabalhador em várias equipes (1:N)
  • Equipe → Trabalhador: Uma equipe possui vários trabalhadores (1:N)
  • Trabalhador → Dirigente: Um trabalhador pode ser dirigente de várias pastas (1:N)
  • Pasta → Dirigente: Uma pasta pode ter vários dirigentes (1:N)
  • Encontro → Palestra: Um encontro possui várias palestras (1:N)
  • Palestra → Palestrante: Uma palestra pode ter vários palestrantes (N:M)
  • Trabalhador → Contribuicao: Um trabalhador pode fazer várias contribuições (1:N)
  • Encontro → MovimentoFinanceiro: Um encontro possui vários movimentos financeiros (1:N)
  • Pessoa1 + Pessoa2 → Casal: Duas pessoas formam um casal (2:1)
  • Encontro → AtividadeEncontro: Um encontro possui várias atividades no cronograma (1:N)
  • Palestra → AtividadeEncontro: Uma palestra pode ser vinculada opcionalmente a uma atividade do cronograma (1:0..1)

7.3. Constraints e Índices

Unicidades:
  • Pessoa: CPF único, Telefone único
  • Casal: Combinação (Pessoa1, Pessoa2) única
  • Encontrista: Combinação (Pessoa, Encontro) única
  • Dirigente: Combinação (Trabalhador, Pasta) única
  • Encontrista: Token de ficha único
  • Usuario: Username único

8. Casos de Uso Principais

UC001 - Realizar Inscrição de Encontrista

Ator Principal: Secretário

Pré-condições:

  • Usuário autenticado com permissão de secretário
  • Encontro ativo cadastrado
  • Pessoa cadastrada no sistema

Fluxo Principal:

  1. Secretário acessa página de inscrição de encontristas
  2. Sistema exibe formulário de inscrição
  3. Secretário seleciona ou cadastra pessoa
  4. Secretário seleciona encontro ativo
  5. Sistema calcula idade automaticamente baseado na data de nascimento
  6. Secretário informa forma de pagamento e valor pago
  7. Secretário seleciona círculo (A-H)
  8. Secretário adiciona observações (opcional)
  9. Sistema valida unicidade (pessoa + encontro)
  10. Sistema gera token único para ficha digital
  11. Sistema registra inscrição com data/hora atual
  12. Sistema exibe mensagem de sucesso

Fluxos Alternativos:

  • FA2: Capacidade máxima atingida → Sistema alerta mas permite inscrição (lista de espera)

Pós-condições:

  • Token de ficha gerado e disponível para impressão

UC002 - Criar Encontro

Ator Principal: Coordenador

Pré-condições: Usuário autenticado com permissão de coordenador

    • Local
    • Capacidade máxima

  5. Sistema valida datas (fim ≥ início)

  7. Sistema exibe mensagem de sucesso

UC003 - Organizar Equipe de Trabalho

Ator Principal: Coordenador / Dirigente

Fluxo Principal:

  1. Ator acessa página de gestão de equipes
  2. Ator cria nova equipe informando:

    • Tipo de equipe
    • Encontro (se específica) ou permanente
    • Datas de início e fim
  3. Sistema registra equipe
  4. Ator acessa lista de pessoas
  5. Ator seleciona pessoas para adicionar como trabalhadores
  6. Para cada pessoa, ator indica se é dirigente da equipe

Ator Principal: Secretário

Pré-condições: Encontrista inscrito no encontro

Fluxo Principal:

9. Regras de Negócio

9.1. Regras de Cadastro

RN001: Uma pessoa só pode possuir um CPF único no sistema. Se informado, deve ser validado segundo algoritmo oficial.
RN002: O telefone de uma pessoa deve ser único no sistema, seguindo o formato (00) 00000-0000 ou (00) 0000-0000.
RN003: A data de nascimento deve ser anterior à data atual e a pessoa deve ter no mínimo 12 anos para participar como encontrista.
RN004: Um casal é formado por exatamente duas pessoas. A combinação (Pessoa1, Pessoa2) deve ser única no sistema.

9.2. Regras de Transferência entre Paróquias

RN005: Quando um seguidor é transferido para outra paróquia, seu status muda para TRANSFERIDO_SAIDA e é automaticamente inativado.
RN006: Seguidores com status TRANSFERIDO_SAIDA não podem ser cadastrados como trabalhadores em encontros desta paróquia.
RN007: Quando um seguidor é aceito vindo de outra paróquia, seu status muda para TRANSFERIDO_ENTRADA e é automaticamente ativado.
RN008: Quando um seguidor retorna à paróquia de origem, seu status muda para RETORNADO e é automaticamente ativado, podendo trabalhar em encontros novamente.
RN009: Todo registro de transferência (saída, entrada ou retorno) gera um registro histórico imutável na tabela transferencia com data, paróquias envolvidas e usuário que registrou.
RN010: Ao registrar um retorno, o sistema identifica automaticamente a paróquia de onde o seguidor está retornando através do último registro de TRANSFERIDO_SAIDA, ordenado por data de solicitação e ID (desempate).

9.3. Regras de Encontro

RN015: A data de fim do encontro deve ser igual ou posterior à data de início.
RN016: O nome padrão do encontro deve seguir formato "Segue-me YYYY" onde YYYY é o ano de realização.
RN017: Apenas um encontro pode estar ativo para inscrições em um determinado período (evitar conflitos de dados).
RN018: O encontro só pode ser finalizado (inativado) após a data de término prevista.
RN019: Atividades do cronograma (AtividadeEncontro) não podem ser cadastradas ou editadas em encontros finalizados (status inativo). Somente a visualização é permitida para encontros passados.

9.4. Regras de Inscrição

RN020: Uma pessoa pode participar apenas uma vez como encontrista em um mesmo encontro (unicidade pessoa + encontro).
RN021: A idade do encontrista é calculada automaticamente com base na data de nascimento e data do encontro.
RN022: O círculo deve ser uma letra entre A e H. Cada círculo comporta idealmente 8-12 encontristas.
RN023: O valor pago na inscrição não pode ser negativo. Pode ser zero em casos de bolsa/isenção.
RN024: Cada encontrista recebe um token único (UUID) para identificação em ficha digital e futuras consultas.
RN025: A faixa etária padrão para encontristas é de 16 a 23 anos. Encontristas fora desta faixa não são bloqueados pelo sistema, porém são sinalizados nos relatórios de Análise Avançada com ícone de atenção (⚠). A coordenação deve avaliar cada caso individualmente.
RN026: Para encontristas menores de 18 anos, recomenda-se preencher os dados de responsável legal (nome e telefone) e marcar a autorização do responsável. O sistema não bloqueia o cadastro sem esses dados, mas os relatórios de conformidade indicam a ausência.

9.5. Regras de Equipe

RN030: Uma pessoa pode ser trabalhador em múltiplas equipes simultaneamente.
RN031: Uma equipe deve ter no mínimo 1 trabalhador.
RN032: Trabalhadores marcados como dirigentes em equipes específicas devem ser também cadastrados na tabela Dirigente vinculados a uma Pasta.
RN033: Equipes podem ser permanentes (sem vínculo com encontro específico) ou temporárias (vinculadas a um encontro).

9.6. Regras Financeiras

RN040: Contribuições de trabalhadores devem ter valor maior que zero.
RN041: Movimentos financeiros do tipo ENTRADA aumentam o saldo; tipo SAIDA diminui o saldo do encontro.
RN042: O saldo do encontro é calculado como: (Inscrições + Contribuições + Entradas) - Saídas.
RN043: Comprovantes financeiros devem ser armazenados e associados ao movimento/contribuição correspondente.
RN044: Movimentos financeiros não podem ser excluídos após 30 dias de lançamento, apenas marcados como estornados.

9.7. Regras de Palestra

RN050: Uma palestra deve ter pelo menos um palestrante associado.
RN051: Palestrantes podem ser do tipo: PESSOA (individual), CASAL ou GRUPO (múltiplas pessoas).
RN052: O horário de palestras não deve conflitar (duas palestras no mesmo horário).

9.8. Regras do Cronograma de Atividades

RN053: Atividades do cronograma só podem ser cadastradas, editadas ou excluídas em encontros com status ativo. Para encontros finalizados, apenas a visualização é permitida.
RN054: Uma atividade do tipo PALESTRA pode opcionalmente ser vinculada a uma palestra cadastrada do mesmo encontro. Quando vinculada, o cronograma exibe automaticamente os dados do palestrante.
RN055: O cronograma de um encontro pode ser visualizado para todos os encontros (ativos e finalizados). Encontros finalizados são identificados com "(Realizado)" na seleção.
RN056: Os tipos de atividade disponíveis são: PALESTRA, DINAMICA, REFEICAO, ORACAO, SILENCIO, INTERVALO, LOUVOR, TESTEMUNHO, ENCERRAMENTO, OUTRO. Cada tipo possui ícone e cor de identificação visual distintos.

9.8. Regras de Segurança

RN060: Senhas de usuários devem ter no mínimo 6 caracteres.
RN061: Senhas devem ser armazenadas com hash BCrypt (nunca em texto plano).
RN062: Após 5 tentativas de login incorretas, a conta deve ser temporariamente bloqueada por 15 minutos.
RN063: Ações críticas (exclusão, alteração financeira, alteração de permissões) devem ser registradas em auditoria.

10. Fluxos de Processo

10.1. Fluxo de Implantação Inicial

Sequência recomendada para primeira implantação do sistema:

  1. Configurar Sistema: Definir propriedades (caminho de fotos, banco de dados, servidor)
  2. Criar Usuário Administrador: Primeiro acesso ao sistema
  3. Cadastrar Tipos de Equipe: Definir categorias de equipes (Cozinha, Cânticos, etc.)
  4. Cadastrar Pastas Organizacionais: Criar estrutura de pastas (Formação, Pastoral, Logística, etc.)
  5. Cadastrar Pessoas: Importar/cadastrar base de pessoas (recomendado: ordem cronológica inversa)
  6. Cadastrar Casais: Relacionar pessoas casadas
  7. Criar Encontro: Cadastrar encontro com datas, local e valores
  8. Criar Equipes: Formar equipes de trabalho do encontro
  9. Cadastrar Trabalhadores: Vincular pessoas às equipes
  10. Cadastrar Dirigentes: Atribuir dirigentes às pastas
  11. Iniciar Inscrições: Processar inscrições de encontristas

10.2. Fluxo de Execução do Encontro

  1. Pré-Encontro (30-60 dias antes):
    • Criar encontro no sistema
    • Formar equipes de trabalho
    • Cadastrar trabalhadores
    • Programar palestras
    • Iniciar recebimento de inscrições
    • Registrar contribuições de trabalhadores
  2. Preparação (15 dias antes):
    • Distribuir encontristas em círculos
    • Gerar fichas digitais
    • Emitir relatórios de equipes
    • Lançar despesas de preparação
  3. Durante o Encontro:
    • Consultar fichas de encontristas
    • Acompanhar programação de palestras
    • Registrar vendas (se aplicável)
    • Lançar despesas em tempo real
  4. Pós-Encontro:
    • Finalizar lançamentos financeiros
    • Gerar relatório de fechamento
    • Marcar encontro como finalizado
    • Gerar relatórios para prestação de contas
    • Realizar backup dos dados

10.3. Fluxo de Inscrição de Encontrista

  1. Candidato manifestatempo interesse em participar
  2. Secretário verifica se candidato já está cadastrado como Pessoa
  3. Se não cadastrado: criar nova Pessoa com dados completos
  4. Se já cadastrado: atualizar dados se necessário
  5. Secretário cria registro de Encontrista vinculando Pessoa + Encontro
  6. Sistema valida unicidade (pessoa não pode estar inscrita 2x no mesmo encontro)
  7. Secretário informa forma de pagamento e valor
  8. Sistema atribui círculo (manual ou automático balanceado)
  9. Sistema gera token único para ficha
  10. Secretário pode imprimir ficha digital imediatamente ou posteriormente
  11. Inscrição confirmada

10.4. Fluxo de Movimentação Financeira

  1. Receitas:
    • Inscrições de encontristas → Registradas automaticamente ao criar Encontrista
    • Contribuições de trabalhadores → Registrar em Contribuição
    • Doações/Outras receitas → Registrar como Movimento Financeiro tipo ENTRADA
  2. Despesas:
    • Todas as despesas → Registrar como Movimento Financeiro tipo SAIDA
    • Classificar por categoria (Alimentação, Material, Transporte, etc.)
    • Anexar comprovantes quando disponíveis
  3. Fechamento:
    • Consolidar todas as receitas
    • Consolidar todas as despesas
    • Calcular saldo final
    • Gerar relatório de fechamento
    • Aprovar fechamento

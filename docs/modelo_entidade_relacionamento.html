📊 Modelo de Dados - Sistema Segue-me

Modelo Entidade-Relacionamento Completo

Versão: 1.0.0 | Data: 14 de Abril de 2026
SGBD: PostgreSQL 12+ | Schema: public

1. Visão Geral do Modelo

O modelo de dados do Sistema Segue-me foi projetado para suportar o gerenciamento completo de Encontros de Jovens, abrangendo desde o cadastro de pessoas até o controle financeiro e logístico dos eventos.

A estrutura é composta por 18 entidades principais com relacionamentos que garantem integridade referencial e evitam duplicações de dados.

2. Entidades Principais

🧑 PESSOA

Descrição: Representa qualquer pessoa física no sistema (encontristas, trabalhadores, palestrantes)

Tabela: pessoa

  • id BIGSERIAL PRIMARY KEY - Identificador único
  • nome VARCHAR(100) NOT NULL - Nome completo
  • cpf VARCHAR(14) UNIQUE - CPF formatado
  • data_nascimento DATE NOT NULL - Data de nascimento
  • sexo CHAR(1) - Sexo (M/F)
  • endereco VARCHAR(200) - Endereço completo
  • telefone VARCHAR(15) UNIQUE - Telefone único no formato (00) 00000-0000
  • email VARCHAR(100) - E-mail
  • escolaridade VARCHAR(50) - Enum: FUNDAMENTAL, MEDIO, SUPERIOR, POS_GRADUACAO
  • sacramento TEXT - Enum: BATISMO, CRISMA, EUCARISTIA (ElementCollection)
  • observacoes TEXT - Observações gerais
  • foto VARCHAR(255) - Caminho da foto
  • data_cadastro TIMESTAMP - Data/hora do cadastro
  • ativo BOOLEAN DEFAULT TRUE - Status ativo/inativo
Regras:
  • CPF e Telefone devem ser únicos quando informados
  • Data de nascimento deve ser anterior à data atual
  • Exclusão lógica via campo "ativo"
💑 CASAL

Descrição: Relacionamento entre duas pessoas casadas

Tabela: casal

  • id BIGSERIAL PRIMARY KEY
  • pessoa1_id BIGINT NOT NULL REFERENCES pessoa(id)
  • pessoa2_id BIGINT NOT NULL REFERENCES pessoa(id)
  • data_casamento DATE - Data do casamento
  • foto VARCHAR(255) - Foto do casal
  • observacoes VARCHAR(255)
Constraint Única: UNIQUE(pessoa1_id, pessoa2_id) - Evita duplicação de casais
🎯 ENCONTRO

Descrição: Representa um evento/retiro de Encontro de Jovens

Tabela: encontro

  • id BIGSERIAL PRIMARY KEY
  • nome VARCHAR(100) NOT NULL - Nome do encontro (ex: "Segue-me 2025")
  • tema VARCHAR(200) - Tema do encontro
  • data_inicio DATE NOT NULL - Data de início
  • data_fim DATE NOT NULL - Data de término
  • local VARCHAR(200) NOT NULL - Local de realização
  • descricao VARCHAR(1000) - Descrição detalhada
  • capacidade_maxima INTEGER - Número máximo de participantes
  • valor_inscricao NUMERIC(10,2) - Valor da inscrição
  • valor_contribuicao_trabalhador NUMERIC(10,2) - Valor da contribuição sugerida
  • ativo BOOLEAN DEFAULT TRUE - Status ativo/finalizado
👥 ENCONTRISTA

Descrição: Participante inscrito em um encontro (Seguimista)

Tabela: encontrista

  • id BIGSERIAL PRIMARY KEY
  • pessoa_id BIGINT NOT NULL REFERENCES pessoa(id)
  • encontro_id BIGINT NOT NULL REFERENCES encontro(id)
  • data_inscricao TIMESTAMP - Data/hora da inscrição
  • forma_pagamento VARCHAR(50) - DINHEIRO, PIX, CARTAO, TRANSFERENCIA
  • valor_pago NUMERIC(10,2) - Valor pago na inscrição
  • trabalhador_id BIGINT REFERENCES trabalhador(id) - Padrinho/madrinha
  • circulo VARCHAR(10) - Círculo atribuído (A-H)
  • idade INTEGER - Idade calculada automaticamente
  • token_ficha VARCHAR(36) UNIQUE - Token UUID para ficha digital
  • observacoes TEXT - Observações pastorais
  • ativo BOOLEAN DEFAULT TRUE
Constraint Única: UNIQUE(pessoa_id, encontro_id) - Pessoa não pode participar 2x do mesmo encontro
🏷️ TIPO_EQUIPE

Descrição: Categoria/tipo de equipe de trabalho

Tabela: tipo_equipe

  • id BIGSERIAL PRIMARY KEY
  • nome VARCHAR(100) NOT NULL - Nome do tipo (ex: Cozinha, Cânticos, Decoração)
  • descricao VARCHAR(255)
👨‍👩‍👧‍👦 EQUIPE

Descrição: Equipe de trabalho do encontro

Tabela: equipe

  • id BIGSERIAL PRIMARY KEY
  • nome VARCHAR(100) NOT NULL - Nome da equipe
  • tipo_equipe_id BIGINT NOT NULL REFERENCES tipo_equipe(id)
  • encontro_id BIGINT REFERENCES encontro(id) - NULL para equipes permanentes
  • descricao VARCHAR(255)
  • data_inicio DATE - Data de formação
  • data_fim DATE - Data de encerramento
  • ativo BOOLEAN DEFAULT TRUE
🔨 TRABALHADOR

Descrição: Voluntário membro de uma equipe

Tabela: trabalhador

  • id BIGSERIAL PRIMARY KEY
  • pessoa_id BIGINT NOT NULL REFERENCES pessoa(id)
  • equipe_id BIGINT NOT NULL REFERENCES equipe(id)
  • observacoes TEXT
Nota: Uma pessoa pode ser trabalhador em múltiplas equipes simultaneamente
📁 PASTA

Descrição: Divisão organizacional que agrupa equipes

Tabela: pasta

  • id BIGSERIAL PRIMARY KEY
  • nome VARCHAR(100) NOT NULL - Nome da pasta (ex: Formação, Pastoral)
  • descricao TEXT
  • ativo BOOLEAN DEFAULT TRUE
👔 DIRIGENTE

Descrição: Coordenador responsável por uma pasta

Tabela: dirigente

  • id BIGSERIAL PRIMARY KEY
  • trabalhador_id BIGINT NOT NULL REFERENCES trabalhador(id)
  • pasta_id BIGINT NOT NULL REFERENCES pasta(id)
  • data_inicio DATE NOT NULL - Início do mandato
  • data_fim DATE NOT NULL - Fim do mandato
  • data_cadastro TIMESTAMP
  • ativo BOOLEAN NOT NULL DEFAULT TRUE
  • observacoes VARCHAR(255)
Constraint Única: UNIQUE(trabalhador_id, pasta_id) - Trabalhador não pode ser dirigente 2x da mesma pasta
🎤 PALESTRA

Descrição: Palestra ministrada durante o encontro

Tabela: palestra

  • id BIGSERIAL PRIMARY KEY
  • titulo VARCHAR(200) NOT NULL - Título da palestra
  • tema VARCHAR(255) NOT NULL - Enum: FE, VOCACAO, FAMILIA, SERVICO, etc.
  • descricao VARCHAR(500)
  • encontro_id BIGINT NOT NULL REFERENCES encontro(id)
  • data_hora TIMESTAMP - Data/hora programada
🗣️ PALESTRANTE

Descrição: Pessoa, casal ou grupo que ministra palestra

Tabela: palestrante

  • id BIGSERIAL PRIMARY KEY
  • tipo_palestrante VARCHAR(255) NOT NULL - PESSOA, CASAL, GRUPO
  • pessoa_id BIGINT REFERENCES pessoa(id) - Para tipo PESSOA
  • casal_id BIGINT REFERENCES casal(id) - Para tipo CASAL
Relacionamento N:M com Palestra: Tabela associativa palestra_palestrante
Para tipo GRUPO: Tabela associativa palestrante_grupo_pessoa relaciona palestrante com múltiplas pessoas
💰 CONTRIBUICAO

Descrição: Contribuição financeira de trabalhador para o encontro

Tabela: contribuicao

  • id BIGSERIAL PRIMARY KEY
  • trabalhador_id BIGINT NOT NULL REFERENCES trabalhador(id)
  • valor NUMERIC(10,2) NOT NULL - Valor contribuído
  • forma_pagamento VARCHAR(50) NOT NULL - DINHEIRO, PIX, CARTAO, TRANSFERENCIA
  • data_pagamento TIMESTAMP
  • comprovante_url VARCHAR(255) - Caminho do arquivo de comprovante
  • observacoes VARCHAR(255)
📊 MOVIMENTO_FINANCEIRO

Descrição: Receita ou despesa do encontro

Tabela: movimento_financeiro

  • id BIGSERIAL PRIMARY KEY
  • encontro_id BIGINT NOT NULL REFERENCES encontro(id)
  • tipo VARCHAR(10) NOT NULL - ENTRADA ou SAIDA
  • categoria VARCHAR(20) NOT NULL - ALIMENTACAO, MATERIAL, TRANSPORTE, etc.
  • data_movimento DATE NOT NULL - Data da movimentação
  • data_criacao TIMESTAMP - Data/hora do registro
  • descricao VARCHAR(255) NOT NULL - Descrição do movimento
  • valor NUMERIC(10,2) NOT NULL - Valor da movimentação
  • responsavel VARCHAR(100) - Responsável pelo lançamento
  • comprovante_url VARCHAR(255)
  • observacoes VARCHAR(500)
🛍️ VENDA_ARTIGO

Descrição: Artigo disponível para venda

Tabela: venda_artigo

  • id BIGSERIAL PRIMARY KEY
  • nome VARCHAR(100) NOT NULL
  • descricao TEXT
  • categoria_id BIGINT REFERENCES categoria_artigo(id)
  • preco NUMERIC(10,2) NOT NULL
  • estoque INTEGER DEFAULT 0
  • foto VARCHAR(255)
  • ativo BOOLEAN DEFAULT TRUE
🧾 VENDA_PEDIDO

Descrição: Pedido de venda realizado

Tabela: venda_pedido

  • id BIGSERIAL PRIMARY KEY
  • encontro_id BIGINT NOT NULL REFERENCES encontro(id)
  • data_pedido TIMESTAMP
  • forma_pagamento VARCHAR(50)
  • valor_total NUMERIC(10,2)
  • status VARCHAR(20) - PENDENTE, PAGO, CANCELADO
  • observacoes TEXT
Relacionamentos:
- venda_item_pedido: Itens do pedido (N:1 com artigo)
- venda_pedido_pessoa: Associação N:M com pessoas compradoras
👤 USUARIO

Descrição: Usuário com acesso ao sistema

Tabela: usuario

  • id BIGSERIAL PRIMARY KEY
  • username VARCHAR(50) UNIQUE NOT NULL - Nome de usuário
  • senha_hash VARCHAR(255) NOT NULL - Hash BCrypt da senha
  • pessoa_id BIGINT REFERENCES pessoa(id) - Pessoa vinculada
  • ativo BOOLEAN DEFAULT TRUE
  • data_criacao TIMESTAMP
📝 AUDITORIA

Descrição: Registro de log de auditoria de ações

Tabela: auditoria

  • id BIGSERIAL PRIMARY KEY
  • usuario VARCHAR(255) - Nome do usuário que executou a ação
  • data_hora TIMESTAMP - Data/hora da ação
  • acao VARCHAR(255) - Tipo de ação (INSERT, UPDATE, DELETE)
  • entidade VARCHAR(255) - Nome da entidade afetada
  • entidade_id BIGINT - ID do registro afetado
  • detalhes TEXT - Detalhes adicionais em JSON

3. Relacionamentos entre Entidades

PESSOA ←→ CASAL (1:N e 1:N)
- Uma pessoa pode estar em vários casais (como pessoa1 ou pessoa2)
- Um casal possui exatamente 2 pessoas
- Constraint: UNIQUE(pessoa1_id, pessoa2_id)
PESSOA → ENCONTRISTA (1:N)
- Uma pessoa pode participar como encontrista em vários encontros
- Constraint: UNIQUE(pessoa_id, encontro_id) - pessoa só participa 1x por encontro
ENCONTRO → ENCONTRISTA (1:N)
- Um encontro possui vários encontristas inscritos
- Cascata: ao excluir encontro, exclui encontristas
PESSOA → TRABALHADOR (1:N)
- Uma pessoa pode ser trabalhador em várias equipes
- O mesmo trabalhador pode estar em equipes diferentes
EQUIPE → TRABALHADOR (1:N)
- Uma equipe possui vários trabalhadores
- Cada registro de trabalhador pertence a uma equipe específica
TIPO_EQUIPE → EQUIPE (1:N)
- Um tipo de equipe pode ter várias equipes
- Cada equipe possui um tipo
ENCONTRO → EQUIPE (1:N, opcional)
- Um encontro pode ter várias equipes
- Equipes podem ser permanentes (encontro_id NULL) ou específicas de um encontro
TRABALHADOR → DIRIGENTE (1:N)
- Um trabalhador pode ser dirigente de várias pastas
- Constraint: UNIQUE(trabalhador_id, pasta_id)
PASTA → DIRIGENTE (1:N)
- Uma pasta pode ter vários dirigentes ao longo do tempo
- Uso do campo "ativo" para identificar dirigentes atuais
ENCONTRO → PALESTRA (1:N)
- Um encontro possui várias palestras
- Cada palestra pertence a um encontro específico
PALESTRA ←→ PALESTRANTE (N:M)
- Uma palestra pode ter vários palestrantes
- Um palestrante pode ministrar várias palestras
- Tabela associativa: palestra_palestrante(palestra_id, palestrante_id)
PALESTRANTE ← PESSOA | CASAL (1:1 condicional)
- Tipo PESSOA: referencia pessoa_id
- Tipo CASAL: referencia casal_id
- Tipo GRUPO: usa tabela palestrante_grupo_pessoa (N:M com pessoa)
TRABALHADOR → CONTRIBUICAO (1:N)
- Um trabalhador pode fazer várias contribuições
- Cada contribuição é de um trabalhador específico
ENCONTRO → MOVIMENTO_FINANCEIRO (1:N)
- Um encontro possui vários movimentos financeiros
- Cada movimento pertence a um encontro
ENCONTRO → VENDA_PEDIDO (1:N)
- Um encontro pode ter vários pedidos de venda
- Cada pedido pertence a um encontro
VENDA_PEDIDO ←→ PESSOA (N:M)
- Um pedido pode ter váriasço pessoas compradoras
- Uma pessoa pode fazer vários pedidos
- Tabela associativa: venda_pedido_pessoa
VENDA_PEDIDO → VENDA_ITEM_PEDIDO (1:N)
- Um pedido possui vários itens
- Cada item referencia um artigo
USUARIO ← PESSOA (1:1 opcional)
- Um usuário pode estar vinculado a uma pessoa
- Uma pessoa pode ter um ou nenhum usuário

4. Índices e Constraints

4.1. Chaves Primárias

Todas as entidades possuem chave primária do tipo BIGSERIAL (auto-incremento).

4.2. Constraints de Unicidade

Tabela Constraint Única Propósito
pessoa UNIQUE(cpf) CPF único quando informado
pessoa UNIQUE(telefone) Telefone único (obrigatório)
casal UNIQUE(pessoa1_id, pessoa2_id) Evitar duplicação de casais
encontrista UNIQUE(pessoa_id, encontro_id) Pessoa participa 1x por encontro
encontrista UNIQUE(token_ficha) Token único para ficha digital
dirigente UNIQUE(trabalhador_id, pasta_id) Trabalhador dirigente 1x por pasta
usuario UNIQUE(username) Nome de usuário único

4.3. Chaves Estrangeiras

Todas as relações entre tabelas são garantidas por chaves estrangeiras com integridade referencial.

4.4. Índices Recomendados

Tabela Coluna(s) Tipo Justificativa
pessoa nome BTREE Consultas por nome (filtros, autocomplete)
pessoa telefone UNIQUE Buscas por telefone (já cria índice)
encontrista encontro_id BTREE Listagem de encontristas por encontro
encontrista circulo BTREE Filtros por círculo
movimento_financeiro encontro_id, data_movimento BTREE Relatórios financeiros por período
auditoria data_hora BTREE DESC Consultas de logs mais recentes

6. Considerações de Performance

6.1. Otimizações Implementadas

  • Lazy Loading: Relacionamentos OneToMany e ManyToMany com FetchType.LAZY
  • Índices em FK: PostgreSQL cria automáticamente índices em chaves primárias e únicas
  • Paginação: Uso de paginação em consultas com grandes volumes (ex: listagem de pessoas)
  • Exclusão Lógica: Campo "ativo" evita exclusões físicas e mantém histórico

6.2. Recomendações para Produção

  • Configurar connection pooling adequado (mínimo 10, máximo 50 conexões)
  • Habilitar query logging em desenvolvimento, desabilitar em produção
  • Executar VACUUM ANALYZE periodicamente para otimizar estatísticas do PostgreSQL
  • Monitorar queries lentas (> 1 segundo) e criar índices adicionais conforme necessário
  • Implementar estratégia de backup diário automático

7. Script de Criação do Banco

O script SQL completo de criação do banco de dados está disponível em:

modelo/modelo_relacional.sql

Inicialização Automática:
O projeto utiliza JPA/Hibernate com hibernate.hbm2ddl.auto=update que cria/atualiza automaticamente o schema do banco baseado nas entidades JPA.

Para ambientes de produção: recomenda-se usar migrations (Flyway/Liquibase) ao invés de auto-update.
